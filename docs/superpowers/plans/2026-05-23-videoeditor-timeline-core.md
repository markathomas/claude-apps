# VideoEditor — Timeline Core Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make VideoEditor genuinely useful — drag clips from the MediaPool onto a real timeline, trim/split/delete them, undo/redo, snap to edges, scrub a playhead that drives the preview off the proxies generated by Plan 2, and render filmstrip thumbnails + audio waveforms on the clips themselves. Definition of done: **a user can cut a real video end-to-end inside the app**, save the project, reopen it, and the timeline (plus the media it references) loads correctly.

**Architecture:** Timeline mutation is implemented as **pure Rust reducer functions** over `Timeline` (no I/O, no `&mut self` on the Tauri side beyond returning the new state) so it's trivially unit-testable and Tauri commands stay thin. The frontend owns a `timelineStore` that wraps `projectStore` and exposes high-level actions (`insertClip`, `moveClip`, `trimClip`, `splitClip`, `deleteClip`); each action calls a Tauri command, receives the new `Timeline`, swaps it into the project, and pushes the previous timeline onto an in-memory undo stack. Snap-to-edge is computed Rust-side during move/trim using a configurable threshold (default 5 frames at the project framerate). Drag-and-drop uses the HTML5 DataTransfer API carrying `application/x-videoeditor-media-id`. Preview playback is driven by a frontend `playheadStore` that ticks via `requestAnimationFrame`, finds the active `VideoClip` at the current playhead position, and seeks the `<video>` element of the clip's proxy to `(playhead - timeline_start_ms + source_in_ms)`. Filmstrip thumbnails and waveforms are rendered as `<canvas>` overlays on each clip using the JPEGs and peak-data already produced by Plan 2.

`media_pool` reconciliation: on `open_project`, the project's `media_pool` field is now populated when items are imported, and on load we re-register every `MediaItem` into the in-memory `MediaRepo` (re-marking `proxy` / `thumbs` / `waveform` paths as `ready` if they exist on disk, or re-queueing proxy generation if they don't).

**Tech Stack:** Rust pure-function reducers (no new crates), Tauri 2 commands + events, Svelte 5 runes + stores, native HTML5 drag-and-drop, `<canvas>` for filmstrip + waveform rendering, `<video>` element driven by `requestAnimationFrame` for preview playback.

- [ ] **Step 1: Write failing tests for `playheadStore`.**

- `activeClipAt(timeline, ms)` returns the clip whose range contains `ms`, or `null`.
- `play()` advances `playheadMs` by `(performance.now delta)` on each tick, capped at `timeline.duration_ms`, then auto-pauses.
- `pause()` stops ticks.
- `seek(ms)` clamps to `[0, duration_ms]` and updates immediately.

- [ ] **Step 2: Implement.**

`requestAnimationFrame` loop while `playing`. Source of truth is a Svelte store; subscribers (PreviewPlayer, Timeline playhead line) react.

- [ ] **Step 3: Modify PreviewPlayer.**

Subscribe to `playheadStore`. Compute `activeClip = activeClipAt(timeline, playheadMs)`:
- If `activeClip` differs from the previously rendered one, swap the `<video>` `src` to that clip's media's proxy blob URL.
- On every playhead change, set `video.currentTime = (playheadMs - clip.timeline_start_ms + clip.source_in_ms) / 1000`.
- When `playing`, call `video.play()`; otherwise `video.pause()`.

This deliberately avoids gap-less concatenation across clips for v1 — there will be a brief reload between clips. That's acceptable for the "I can cut a video" milestone; gapless preview is a follow-up.

## What ships at the end of this plan

- Drag clips from the **MediaPool** onto a real **Timeline**
- Move, trim (head + tail), split, and delete clips on the timeline
- Snap to neighboring clip edges and to timeline 0
- Undo / redo with Cmd/Ctrl+Z and Cmd/Ctrl+Shift+Z
- Preview playback driven by a timeline playhead — space to play/pause, scrub by clicking the ruler
- Filmstrip thumbnails on each clip, scaled to its trimmed source range
- Audio waveform on each clip (or a `no audio` tag when absent)
- Saving and reopening a project preserves the timeline **and** rehydrates the media pool from disk

**Test counts at end of plan (cumulative):**
- Rust: ~85 unit tests + ~8 integration tests
- Frontend: ~30 Vitest tests across ~6 files

**Definition of done:** a user can cut a real video end-to-end inside the app, save, reopen, and keep editing.

## What's NOT in this plan (deferred)

- **Export** (rendering the timeline to a final MP4). Lands in Plan 4.
- **Transitions visualization** on the timeline. Transitions exist in the model but are not yet rendered or applied during preview.
- **Multi-track video** (overlays, picture-in-picture). Single video track + single audio track only.
- **Audio mixing UI** (per-clip volume drag, fade handles on the waveform). Mixer lands in Plan 4 alongside export.
- **Gapless preview across clips.** Clip transitions during playback briefly reload the `<video>`. Acceptable for v1.
- **Text/title clip editing UI.** The text track is in the model and saved/loaded but has no editor.
- **Drag-and-drop import from the OS** (still file picker only).
- **Toast/error UX.** Failed timeline ops log to DevTools console; modal error UX lands in Plan 7.
- **Zoom-to-fit, fit-track-to-window.** Manual zoom only via a slider.
- **Keyboard shortcuts beyond the minimum** (no marquee selection, no ripple delete, no slip/slide trim).

## Next plan

**Plan 4 — Export pipeline.** Wire up an FFmpeg-driven export that renders the assembled timeline to a single MP4 file: concat per-track segments, apply trims and transitions, mix audio, write progress to a modal. With Plan 4, VideoEditor stops being an editor-without-output and becomes a tool you can actually use to produce a finished video.
